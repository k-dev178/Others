## 4. 서버 & 배치 업무 상세

### 4-1. 배치(Batch)란?

- **정의**: 여러 작업을 **“한 번에 몰아서 처리”**하는 것
- 예시:
  - 고객이 물건을 살 때마다 택배사에 실시간 호출 X
  - 일정 시간마다 물류 정보/택배 요청을 모아서 한 번에 전송 → 비용/효율 최적화
- 이커머스에서 배치가 하는 일:
  - 주문/배송 정보 일괄 처리
  - 정산, 포인트, 쿠폰 정리
  - 각종 대량 데이터 처리 업무

### 4-2. 배치 관련 본인 역할

- 배치 전용 서버 구축 및 모니터링
- 기존 유료 배치 솔루션 → **직접 개발한 배치 시스템**으로 교체
  - 회사가 내는 솔루션 비용 절감
- 실패 시 위험:
  - 예: 배송지 계산 잘못 → **수천 건 오발송** → 모든 비용은 회사 부담
- 그래서:
  - 배치 실행 시 온전히 집중
  - 이어폰 끼고, 사람 말 거의 안 듣고 수행

### 4-3. 서버 모니터링 & 장애 대응

- 모니터링 화면에서:
  - 서버별 상태, 트래픽, 쿼리 응답 속도 등 확인
  - 이상 시 빨간색 경고 뜸 → 즉시 조치
- 디스크 사용량 예시:
  - 96% 사용 중이면 **97% 되기 전에** 로그 정리/삭제
  - 100% 되면 서비스 응답 불가 → 장애
- 실제 사례:
  - 설날 아침에 서버 Out of Memory → 컨테이너 다운
  - 고향(전주) 내려가 있는 상황에서 비상 연락 받고 긴급 재시작/조치

### 4-4. U2L (Unix → Linux) 프로젝트

- U: Unix, L: Linux, “from Unix to Linux”
- 이유:
  - Unix(AIX, macOS 계열 등)는 **유지보수/라이선스 비용이 매우 비쌈**
  - Linux는 무료이며, 성능도 많이 발전함
- 고려해야 할 것들:
  - OS/IP 변경에 따른 방화벽, 스토리지, 미들웨어, 로드밸런서 설정
  - 자바/미들웨어 버전 호환
  - 애플리케이션 재배포

## 5. 백엔드 개발자의 관점: 웹/서버 개념

### 5-1. 웹 서버 vs WAS(미들웨어)

- **웹 서버**
  - 정적 리소스(HTML, 이미지 등) 제공
  - 누구나 같은 URL에 접속하면 같은 화면을 보는 부분
  - 예: 학교 홈페이지 메인, 공지사항 목록 등
- **WAS(웹 어플리케이션 서버, 미들웨어)**
  - 로그인 등 **사용자별로 다른 화면**을 제공
  - 예: 마이페이지, 수강 신청 내역, 성적 조회
  - 동적 리소스를 제공하는 영역

### 5-2. DNS & 인터넷 동작 개요

- 사용자가 `naver.com`, `jj.ac.kr` 입력
- 컴퓨터는 문자 주소를 이해 못함 → **DNS 서버에게 물어봄**
  - “이 도메인의 IP 주소가 뭐냐?”
- 국내에는 ISP(KT, SKT, LG U+)가 관리하는 DNS 서버들이 있음
- DNS가 IP를 알려주면 해당 서버로 접속
- 브라우저가 HTML/JS/CSS 렌더링 → 화면에 페이지 표시

> 이 정도 원리를 설명할 수 있으면 백엔드 로드맵의 **첫 단계는 통과**했다고 보는 느낌으로 설명.

## 8. 코딩 테스트 & 알고리즘

### 8-1. 코딩 테스트의 현실

- 최근 GPT 등 때문에 **온라인 코테를 줄이는 추세**는 있음 (확실하지 않음)
- 하지만 여전히 많은 회사에서 코테를 시행 중
- 특히 신입/주니어 채용에서는 거의 기본 필터 역할

### 8-2. 코테 준비 방법

- 언어는 **회사에서 사용하는 주력 언어** 기준으로
  - 백엔드(Java) → Java로 푸는 게 자연스러움
  - Node.js 뽑으면 JS/TS, Python 뽑으면 Python 등
- **알고리즘이 핵심**:
  - 언어는 도구일 뿐, 알고리즘/자료구조 이해가 중요
- **라이브러리 사용 제한**
  - 표준 라이브러리는 OK
  - `pip install`, `npm install` 해야 쓰는 **외부 라이브러리**는 거의 모두 사용 불가
  - 파이썬의 `numpy` 같은 건 못 쓴다고 생각해야 함
- 추천 사이트
  - 백준, 프로그래머스, 리트코드 등
- 연사의 루틴
  - 지금도 **하루 한 문제** 정도는 계속 푼다고 언급
  - 예전에 한동안 놀다가 다시 꾸준히 시작했다고 함

### 8-3. “알고리즘이 현업에서 도움이 되나요?”

- **직접적으로는** 대부분의 업무에서 깊은 알고리즘을 쓰는 경우는 드묾
- 하지만:
  - `ArrayList` vs `LinkedList` 선택
  - 자료구조/시간복잡도를 모른 채 코드 짜면 **성능/유지보수에 큰 문제** 발생
  - 코드 리뷰하는 회사에서는 본인이 모르는 부분이 그대로 드러나서 **독**이 될 수 있음
- 결론:
  - “코테만을 위한 공부”로 끝내면 아깝고,
  - 적어도 사용하는 자료구조/알고리즘의 **이유와 복잡도**는 알고 있어야 한다고 강조

## 11. 채용 시장 현실 & 조언

### 11-1. AI 이후 채용 시장 변화

- 경영진들은:
  - 비용 절감을 위해 “AI로 개발자를 대체”하는 상상을 많이 함
- 실제 개발자 입장에서:
  - Cursor, Cloud Code, Gemini CLI 등 **개발자용 AI 에이전트**를 써 봤지만
  - 아직 만족도 높지 않음, “AI 똥 치우는 개발자”라는 새로운 직업이 생길 수도 있다고 농담
- 채용 시장:
  - 특히 **판교** 기준으로 신입 공채가 확연히 줄었다고 느낌
  - 상반기에는 한 자릿수 수준 얘기까지 나올 정도라고 언급(**체감 기반, 확실하지 않음**)
  - 경쟁 상대:
    - 동기, 선배, 후배, 타학교 출신, 경력직 ‘중고 신입’까지 모두 포함

### 11-2. 휴식에 대한 조언

- 본인은 졸업 전에 취업해서 **“학생과 사회인 사이 휴식기가 거의 없었다”**고 회상
- 공부를 충분히 했다면:
  - 졸업 전/후 적당한 휴식도 마지막으로 쉴 수 있는 중요한 시간일 수 있다고 조언
- 단, 전제:
  - “충분히 공부했다”는 자기 기준이 필요

### 11-3. 개발자 성향 체크

- 개발자 일은:
  - 야근, 긴급 장애, 책임감 등으로 인해 체력/멘탈 소모가 큼
- **성격이 안 맞으면 진짜 고통**일 수 있음
- 반대로 코드/시스템 설계 자체가 재밌으면:
  - 힘들어도 버티면서 실력 쌓이고 성장 가능
